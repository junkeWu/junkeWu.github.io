Map
2022-05-19 10:28:01 # golang

特点

  • map是一种key，value存储结构。
  • key通常为int、string；value通常为数字（整数、浮点数）、string、map，结构体。
  • key：slice、map、function不可以。
  • map类型的变量默认初始值为nil，需要使用make()函数来分配内存。
  • 键不能重复
  • 键必须可hash (int/bool/float/string/array)
  • 无序

Make map

make(map[keyType]valueType, cap)

其中cap表示map的容量，该参数虽然不是必须的，但是我们应该在初始化map的时候就为其指定一个合适的容量。

value, ok := map[key] // 判断建是否存在

初始化

  • 第一步: 创建一个hamp结构体对象
  • 第二步：生成一个哈希因子hash0并赋值到hmap对象中（用于后续为key创建哈希值）。
  • 第三步：根据hint=10，并根据算法来创建B
  • 第四步：根据B去创建桶并存放在buckets数组中
    • 当B < 4 时，根据B创建的桶的个数的规则为： 2^B (标准桶)
    • 当B >=4 时，根据B创建桶的个数的规则为：2^B + 2^B-4（标准桶+溢出桶）

每个bmap可以存储8个键值对，当不够存储时需要溢出桶，并将当前bmap的overflow字段指向溢出桶的位置。

原码

写入数据

info ["name"]="junkewu"

在map中写入数据，内部执行流程为：

  • 第一步：结合哈希因子和键name生产哈希值01101110001111110111011011
  • 第二步：获取哈希值的后B位，并根据后B位的值来决定将此键值对存放的那个桶中（bmap）。
  • 第四步：hamp的个数count++（map）中的元素个数+1

读取数据

value := info["name"]

在map中读取数据流程：

  • 第一步： 结合哈希因子和键name生成哈希值。
  • 第二步：获取哈希值的后B位，并根据B位的值来决定此键值对存放在哪个桶（bmap）中。
  • 第三步：确定桶之后，再根据key的哈希计算出tophash（高8位），根据tophash和key去桶中查找数据。

当前桶如果没有找到，则根据overflow在去溢出桶中找 ，均未找到则表示key不存在。

当写入数据时，key值hash重复的情况下，会存储到overflow溢出桶中，使用拉链法的形式处理，tophash是唯一的。

当读取数据时，会先遍历tophash数组，找到相应的位置，依此知道key、value的位置。

扩容

当向map中添加数据时，当达到某个条件，则会引发字段扩容。

触发扩容条件

  • map中数据总个数/桶个数 > 6.5，引发翻倍扩容。
  • 使用了太多的溢出桶时（溢出桶使用的太多会导致map处理速度降低）。
    • B <= 15,已使用的溢出桶个数 >= 2^B时，引发等量扩容。
    • B > 15, 已使用的溢出桶个数 >= 2^15时，引发等量扩容。

  • 第一步：B会根据扩容后的新桶的个数进行增加（翻倍扩容新B=旧B+1，等量扩容新B=旧B）
  • 第二步：oldbuckets指向旧桶
  • 第三步：buckets指向新创建的桶（新桶中暂时还没有数据）。
  • 第四步：nevacuate设置为0，表示如果数据迁移的话，应该从原桶中的第0个位置开始迁移。
  • 第五步：noverflow设置为0，扩容后新桶中已使用的溢出桶为0.
  • 第六步：extra.oldoverflow设置为原桶已使用的所有溢出桶。即：h.extra.oldoverflow=h.extra.overflow
  • 第七步: extra.overflow设置为nil，因为新桶还未使用溢出桶。
  • 第八步：extra.nextOverflow设置为新创建的桶中的第一个溢出桶的位置。

迁移

  • 迁移时会遍历某个旧桶中的所有的key（包括溢出桶），并根据key重新生产哈希值，根据哈希值低8位来决定将此键值分流到新桶中。

扩容后，B的值在原来的基础上加1，也就意味着通过多1位来计此键值对要分流到新桶位置，如上图：

  • 当新增的位（红色）的值为0，则数据会迁移到与旧桶与旧桶编号一致的位置。
  • 当新增的位（红色）的值为1，则数据会迁移到翻倍后对应编号位置。

等量扩容

如果是等量扩容（溢出桶太多引发的扩容），那么数据迁移机制就比较简单，就是讲旧桶（含溢出桶）中的值迁移到新桶中。这种扩容和迁移的意义在于：当溢出桶比较多而每个桶中的数据又不多时，可以用过等量扩容和迁移让数据更紧凑，从而减少溢出桶。

非线程安全

map默认是并发不安全的，同时对map进行并发读写时，程序会panic，原因如下：

Go官方在经过长时间讨论后，认为Go Map更应适配典型使用场景（不需要从多个goroutine中进行安全访问），而不是为了小部分情况（并发访问），导致大部分程序付出加锁代价（性能）。

解决非线程安全

  • 读写锁 sync.RWMutex
  • sync.map
    • （空间换时间），减少加锁对性能的影响，可以无锁访问read map，而且会优先操作read map，倘若只操作read map就可以满足要求，那就不用去操作write map（dirty），所以在某些特定场景中它发生锁竞争的频率会远远小于map + RWMutex的实现方式。
    • 适合读多写少的场景。
    • 写多的场景，会导致read map 缓存失效，需要加锁，冲突变多，性能急剧下降。

都是解决读写分离。

Map注意事项

  • map无论有没有进行扩容，指向都是同一个地址。
